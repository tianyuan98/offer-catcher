## 修复：AI分析胡说八道（岗位类型识别+假数据清理）

### 用户反馈的两个截图问题

1. **招聘14人被质疑为"中风险"，追问GitHub链接** —— HR的正常工作成果被当成技术项目数据质疑
2. **HR简历被建议"补充运营数据"** —— 简历里出现了"运营"两个字，就自动触发"活动运营缺少数据化"建议

### 问题根因（双层bug）

**第一层：AI prompt没有岗位类型识别**
- 5个AI请求的prompt都没有要求AI先判断岗位类型
- 真实性验证prompt虽然写了"不要质疑基本信息"，但没写"招聘14人对HR来说是正常的"
- 优化建议prompt没有禁止"用运营岗标准评判HR岗"

**第二层：本地analyzer.ts全是硬编码假数据**
- `generateOptimizationSuggestions()`：只要简历含"运营"或"活动"就自动生成"活动运营缺少数据化"建议（完全不JD是什么岗）
- `extractClaims()`：用正则 `/\d+.*?（提升|增长|降低|优化|完成|实现）/` 匹配任何含数字+中文的句子，"14名适配人员...有效降低"被误匹配
- `generateOptimizedResume()`：永远返回同一套假简历内容
- `analyzeMatch()`：永远返回72分假匹配数据
- `generateInterviewPrep()`：永远返回6道数据运营面试题
- 当AI请求失败时，page.tsx fallback到这些假数据

### 修复内容

**1. 重写全部5个AI prompt（page.tsx）**
每个prompt都增加：
- **强制步骤1**：先识别JD对应的岗位类型
- **强制步骤2**：按岗位类型给出判断（HR岗的正常标准 ≠ 技术岗的正常标准）
- **绝对禁止清单**：禁止用A岗标准评判B岗、禁止建议不相关的内容

特别是真实性验证prompt增加了：
- "HR岗：招聘10-20人是正常工作量，不应质疑"
- "证据类型必须匹配岗位：不要给HR岗要GitHub链接"
- "如果简历中没有值得质疑的声明，claims可以为空数组。不要硬找问题。"

**2. 清空analyzer.ts所有假数据**
- `extractClaims()` → 返回空数组
- `generateOptimizedResume()` → 返回空数组
- `analyzeMatch()` → 返回空数据（overallScore=0, dimensions=[]）
- `generateInterviewPrep()` → 返回空数组
- `generateOptimizationSuggestions()` → 只保留ATS关键词缺失提示（基于真实扫描），删除"运营"关键词触发的假建议
- `generateOptimizedTextFromChat()` → 直接返回用户输入内容
- `verifyTruth()` → 无声明时返回安全默认值

**3. 修复fallback逻辑（page.tsx）**
- AI失败时，`optimizedResume` 和 `suggestions` 设为空数组，不再fallback到本地假数据

构建并部署成功。

## 修复：真实性验证质疑个人基本信息问题

### 问题根因
`analyzer.ts` 中的 `extractClaims()` 使用正则匹配数字，把含数字的任何文本（包括"2024-09~至今 中山大学"这种教育背景）都当成"量化声明"质疑，追问"数据来源和计算方法"。

### 修复方案
在 `handleAnalyze` 中新增第5个 AI 请求（并行），用 AI 智能识别哪些是"真正值得质疑的声明"，并在 prompt 中明确规定：
- 不质疑教育背景、实习时间、学校名称等基本事实
- 不质疑个人信息（姓名、学历、专业等）
- 只针对量化成果、技能声明、团队管理声明进行追问

构建并部署成功。线上已更新。

## 修复：简历优化官 AI 结果与简历无关

### 问题根因
`lib/engines/analyzer.ts` 中 `generateOptimizedResume()`、`analyzeMatch()`、`generateInterviewPrep()` 全部使用硬编码假数据，无论用户上传什么简历，"优化对比"、"匹配分析"、"面试准备"三个面板永远显示同一套内容。

### 修复方案
在 `app/resume-optimizer/page.tsx` 的 `handleAnalyze()` 中，改为并行发起 4 个 AI 请求：
1. **优化建议**（含引导问题）：prompt 明确要求基于简历原文，不得使用模板
2. **优化对比**：AI 从简历原文提取真实内容作为 original，生成优化版本
3. **匹配分析**：AI 基于真实简历分析各维度匹配度
4. **面试准备**：AI 基于简历真实经历提问，生成个性化面试题

用 `Promise.all()` 并行执行，结果替换 `analysisResult.optimizedResume`、`analysisResult.matchReport`、`analysisResult.interviewPrep`。

### 关键原则
- AI prompt 中明确注明"原始内容必须直接来自简历原文，不得编造"
- 4 个请求并行，减少用户等待时间
- 本地引擎仍用于 ATS 关键词匹配（速度快），AI 专注于需要理解能力的模块

## 修复：Word文档无法解析 + AI建议太泛泛

### 问题1：Word文档上传报错
**表现**：上传 .docx 文件提示 "Word 文件暂不支持自动解析"
**根因**：`lib/pdf-extract.ts` 中对 .doc/.docx 直接返回错误，没有解析逻辑
**修复**：安装 `mammoth` 库，在 `extractTextFromFile` 中添加 .docx 解析逻辑（读取 ArrayBuffer → mammoth.extractRawText）

### 问题2：AI建议没有针对性
**表现**：建议只有 "补充项目成果的数据指标"、"在技能栏突出JD要求的核心技能" 等模板化内容
**根因**：
1. AI prompt 没有强制要求引用简历原文
2. 7B模型默认会输出通用简历建议模板
3. 没有给模型展示"什么是好的建议"vs"什么是差的建议"

**修复**：重写全部5个AI prompt + 对话优化prompt，增加：
1. **8项核心分析框架**：关键词匹配、STAR法则、量化数据、动词强度、胜任力呈现、结构逻辑、ATS友好、表述合规
2. **正反示例（Few-shot）**：明确展示错误示例（泛泛而谈）和正确示例（引用原文+具体修改）
3. **强制引用原文**：每条建议必须指出简历中的具体句子/模块/位置
4. **岗位类型特定维度**：技术岗/产品岗/运营岗/HR岗等各有一套分析维度
5. **更强的禁止清单**：明确禁止输出没有引用原文的建议
